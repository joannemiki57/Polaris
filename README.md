# Polaris — MindGraph

An interactive research exploration tool that generates knowledge graphs from academic questions. Users ask a question, the system retrieves real papers from [OpenAlex](https://openalex.org) and uses an LLM to organize their topics into a navigable mind map, then lets users selectively expand, attach papers, and generate deep-dive answers — all rendered as a React Flow canvas.

┌─────────────────────────────────┐

│  React + React Flow canvas      │
│  Question input → Graph UI      │
│  Node selection → Deep panel    │
└──────────┬──────────────────────┘
           │  /api/* (proxied by Vite)
           ▼
┌─────────────────────────────────┐
│       Express BFF (8787)        │
│  ┌───────────┐  ┌────────────┐  │
│  │  llm.ts   │  │ openalex.ts│  │
│  │  OpenAI / │  │ Search,    │  │

The server acts as a Backend-for-Frontend (BFF): it holds API keys, orchestrates calls to OpenAlex and the LLM, merges graph deltas, and returns complete `MindGraph` JSON to the client.

## Data model

Defined in `graphTypes.ts` (shared between client and server).

### Node kinds

| Kind | Description |
|------|-------------|
| `topic` | Root-level research topic derived from the user's question |
| `keyword` | Research concept extracted from papers or generated by LLM |
| `subtask` | Actionable research sub-task |
| `paper` | Academic paper from OpenAlex, carries `openAlexId`, `doi`, citation count |
| `note` | User-created annotation |

### Edge kinds

| Kind | Description |
|------|-------------|
| `expands_to` | Parent concept → child concept (hierarchical) |
| `prerequisite_for` | Concept A is required knowledge for concept B |
| `from_openalex` | Topic/keyword → paper (search result linkage) |
| `has_keyword` | Paper → keyword (extracted from OpenAlex metadata) |
| `user_linked` | Manual user-created connection |

### MindGraph schema

```typescript
interface MindGraph {
  version: 1;
  title: string;
  nodes: GraphNode[];
  edges: GraphEdge[];
  updatedAt: string;
}

interface GraphNode {
  id: string;
  kind: NodeKind;
  label: string;
  summary?: string;
  openAlexId?: string;
  doi?: string;
  year?: number;
  citedByCount?: number;
  url?: string;
  relevance?: number;
  isReview?: boolean;
}
```

## API routes

All routes are served from Express on port `8787`. The Vite dev server proxies `/api/*` requests.

| Method | Path | Rate limit | Description |
|--------|------|-----------|-------------|
| `GET` | `/api/health` | default | Returns LLM availability and OpenAlex config |
| `POST` | `/api/graph/expand` | 60/min | **Initial graph**: question → hybrid OpenAlex + LLM pipeline |
| `POST` | `/api/graph/expand-selection` | 60/min | Expand selected nodes via LLM |
| `POST` | `/api/graph/expand-paper-keywords` | 60/min | Expand a paper node with its OpenAlex keywords |
| `POST` | `/api/graph/attach-papers` | 60/min | Search OpenAlex and attach paper nodes to a keyword |
| `POST` | `/api/llm/deep` | 20/min | Generate a deep markdown answer for selected nodes |
| `GET` | `/api/openalex/works` | 20/min | Raw OpenAlex search passthrough |

## Core pipelines

### 1. Hybrid initial expansion (`POST /api/graph/expand`)

The main graph generation pipeline. Produces a literature-grounded knowledge graph rather than relying on pure LLM invention.

```
User question
    │
    ├──→ OpenAlex: search 3 review papers (type:review, relevance-sorted)
    ├──→ OpenAlex: search 3 top-cited articles (type:article, relevance-sorted)
    │
    ├──→ For each paper: fetch topics, keywords, and abstract
    │
    └──→ LLM (organizeKeywordsToGraph):
         "Organize these REAL keywords into a tree.
          Do NOT invent new keywords."
              │
              ▼
         MindGraph with:
         - Topic nodes (organized hierarchy)
         - Paper nodes (with OpenAlex IDs, DOIs, citation counts)
         - Review papers flagged with isReview: true
```

The LLM receives paper titles, OpenAlex topics (with subfield/field hierarchy), and the first 400 characters of each abstract. It structures the real keywords — it does not hallucinate new ones.

### 2. Selection expansion (`POST /api/graph/expand-selection`)

When the user selects nodes and clicks "Expand Selected":
- **Paper nodes** with `openAlexId` → routed to the paper keyword expansion pipeline (no LLM)
- **Non-paper nodes** → LLM generates new child nodes and merges them via `mergeDelta`

### 3. Paper keyword expansion (`POST /api/graph/expand-paper-keywords`)

Data-driven, no LLM required:

```
Paper node (has openAlexId)
    │
    └──→ OpenAlex: GET /works/{id}?select=id,display_name,keywords
              │
              ▼
         Keyword nodes + has_keyword edges
         (sorted by relevance score, capped at ~15)
```

### 4. Paper attachment (`POST /api/graph/attach-papers`)

Search OpenAlex for a query string and attach matching papers to a keyword node as `paper` nodes with `from_openalex` edges.

### 5. Deep answer (`POST /api/llm/deep`)

Given a question and optionally selected node labels, generates a long-form markdown analysis. Rate-limited more aggressively (20/min).

## OpenAlex integration

`server/src/openalex.ts` wraps the [OpenAlex API](https://api.openalex.org) with:

| Function | Purpose |
|----------|---------|
| `searchWorks` | Full-text search with optional type filter (`review` / `article`). Uses relevance sorting for filtered queries, citation sorting for unfiltered. |
| `fetchWorkDetail` | Fetches topics, keywords, and reconstructed abstract for a single work |
| `fetchWorkKeywords` | Convenience wrapper returning just keywords with relevance scores |
| `workHitToPaperNodes` | Converts search results into `GraphNode[]` with proper IDs and metadata |
| `keywordsToGraphNodes` | Converts keyword arrays into nodes + `has_keyword` edges |
| `reconstructAbstract` | Rebuilds readable text from OpenAlex's `abstract_inverted_index` format |

**Caching**: Disk-based JSON cache at `server/.cache/openalex.json` with 24-hour TTL. Prevents redundant API calls during development and exploration.

**Rate compliance**: Uses `mailto` header for OpenAlex polite pool access. Single-work lookups are free and unlimited.

## LLM integration

`server/src/llm.ts` handles all LLM interactions via the OpenAI SDK:

| Function | Purpose |
|----------|---------|
| `organizeKeywordsToGraph` | Takes real paper data (titles, topics, abstracts) and produces a structured `MindGraph`. Prompt explicitly forbids inventing new keywords. |
| `expandQuestionToGraph` | Fallback: pure LLM graph generation when OpenAlex data is unavailable |
| `expandFromSelection` | Generates delta nodes/edges from selected concepts |
| `mergeDelta` | Deduplicates and merges new nodes/edges into an existing graph |
| `deepAnswer` | Generates long-form markdown analysis |

The LLM produces structured JSON matching the `MindGraph` schema. Responses are parsed and validated before returning to the client.

## Client architecture

### App structure

`App.tsx` is the main component with three areas:

1. **Sidebar** — Question input, generate button, selection actions (expand, deep answer, attach papers), session management (export markdown, clear)
2. **Canvas** — React Flow graph with custom `MindNode` components, minimap, and controls
3. **Deep panel** — Renders markdown deep-dive answers

### Custom node rendering (`MindNode.tsx`)

Nodes are styled by `kind` with distinct colors/shapes. Paper nodes show citation counts; review papers get a visual badge. Keyword nodes from OpenAlex display relevance scores.

### Layout (`layout.ts`)

Converts `MindGraph` (the domain model) into React Flow's `Node[]` and `Edge[]` format, applying positioning and edge styling by kind.

### Persistence (`persistence.ts`)

Session state (current graph) is saved to `localStorage` so users can resume exploration across page reloads.

### API client (`api.ts`)

Typed fetch wrappers for every server endpoint. All calls go through a shared `j()` helper that handles JSON serialization and error handling.

## Key design decisions

### Hybrid LLM + data pipeline over pure LLM generation

The original approach was 100% LLM — Gemini invented keywords from training knowledge. This produced hallucinated topics disconnected from real literature. The hybrid pipeline fetches real papers and keywords from OpenAlex first, then uses the LLM only to **organize** them into a coherent tree.

### Relevance-first search over citation-count sorting

Sorting by `cited_by_count:desc` returned globally popular papers unrelated to the query (e.g., "Radiomics" for a "federated learning" search). Switching to OpenAlex's default relevance sorting for type-filtered queries fixed this while still returning well-cited papers.

### Topics + abstracts over generic keywords

OpenAlex `keywords` are discipline-level labels ("Computer Science", "Engineering") that lack research specificity. OpenAlex `topics` are hierarchical research concepts (domain > field > subfield > topic) that provide real insight. The pipeline now sends topics and abstract excerpts to the LLM.

### Same button, smart routing

"Expand Selected" detects node type: paper nodes with `openAlexId` trigger data-driven keyword expansion (no LLM call), while other nodes route through LLM expansion. No extra UI needed.

### BFF pattern

The server holds API keys, enforces rate limits, and orchestrates multi-step pipelines (search → fetch details → LLM organize → merge). The client stays simple — it sends a question and gets back a complete graph.

## Known limitations and risks

Detailed analysis in [`docs/limitations.md`](docs/limitations.md).

| Area | Risk |
|------|------|
| **Scope** | The vision bundles semantic search, graph reasoning, long-term memory, and vault integration — each non-trivial alone |
| **Controlled distance** | No operational criteria yet for "fruitful analogy" vs. noise in cross-domain edges |
| **Graph UX** | Visual graphs scale poorly in the head; large graphs become "pretty pictures" without filters, summaries, and progressive disclosure |
| **Trust boundaries** | Vault integration (read/write scope, provenance, permissions) is unspecified |
| **LLM cost** | Graph-guided LLM calls can be expensive; depth/width/token bounds needed |

### Not yet implemented

- LLM re-ranking of keywords against the user's question
- Topics hierarchy expansion (domain > field > subfield > topic)
- Recursive expansion (keyword → search works → expand those works' keywords)
- Keyword deduplication beyond ID matching (synonym merging)
- User controls for review vs. article ratio
- Vault/MCP integration
